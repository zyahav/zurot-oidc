## Architectural Violations Resolved ✅

### Root Cause Identified
- Code pack violations were correctly identified in Session 2
- Architecture (frozen) takes precedence over code pack
- `userId` in profiles schema is for human→profile linking (allowed)
- Mutations should derive `userId` from auth context, not accept as parameter

### Fixes Applied

#### 1. Fixed "No mutation accepts userId as owner" ✅
- **Before**: `createProfile(args: { userId, handle, ... })`
- **After**: `createProfile(args: { handle, displayName, role })` + auth context
- **Before**: `listProfilesForUser(args: { userId })`
- **After**: `listProfilesForUser(args: {})` + auth context

#### 2. Fixed "All data ownership uses profileId" ✅
- **Verified**: Schema already correct (activities.ownerProfileId, permissions.profileId)
- **Verified**: profiles.userId is for human→profile linking only (allowed by architecture line 34)
- **Verified**: No app resources use userId for ownership

### Implementation Details

**convex/profiles.ts changes:**
- `createProfile` now gets userId via `ctx.auth.getUserIdentity()`
- `listProfilesForUser` now gets userId via auth context
- Both mutations no longer accept userId as parameters
- Proper error handling for unauthenticated contexts
