Master Handoff - Draft v0.1

Dorit: Central Operations Router, Visual Console, and Deterministic Workflow Engine

Created for Zuriel / ZurOt. Purpose: bring together the Dorit idea across WhatsApp, Telegram, Phone, ZurOt Hub, PayPal, CRM, leads, calendar, and Facebook/Meta operations.

Core decision: The product name is Dorit. Dorit is not just a Telegram bot. Dorit is the operational center: people speak naturally, Dorit translates the request into a typed command proposal, deterministic code validates and executes, and every step appears visually on the operations screen.

1. Product definition

Dorit is a central admin/operator layer for ZurOt and later other schools or service businesses.

Dorit receives natural language from people through any channel, identifies who they are, loads only the relevant context, routes the request to the correct module, executes only deterministic approved workflows, and writes a visible audit trail.

Channels Dorit can receive from

WhatsApp
Parents, teachers, coordinators, existing bridge where available.
Telegram
Preferred internal pilot channel because bot API and message IDs are more deterministic.
Web chat
Dorit web surface at dorit.zurot.org.
Phone app
Teachers use the calling app, likely phone.zurot.org, and Dorit sees every login/call/outcome event.

Domains Dorit can route to

CRM / All-Contacts PayPal invoices Leads Phone / calling Calendar Facebook / Meta ads and leads Teacher availability Parent/student support

2. Non-negotiable architecture principle

The LLM is not the operator.
The LLM is an untrusted translator from natural language into a typed command proposal. Deterministic code validates, authorizes, checks state, asks approval when needed, and executes. 
Human natural language
  -> Channel adapter normalizes message
  -> Identity resolver links sender to ZurOt profile
  -> Context builder loads only relevant context
  -> LLM proposes typed command JSON
  -> Schema validation
  -> Entity resolution
  -> Permission and ownership check
  -> State machine validation
  -> Idempotency check
  -> Approval gate if risky
  -> Deterministic workflow execution
  -> Reply/delivery proof
  -> Visual audit event trail

3. Why Dorit solves the WhatsApp headache

The old problem was hidden state: messages could arrive, cursors could move, processes could run, but the real outcome was unclear. Dorit must make the real contract visible:

  • Every inbound message has a stable event id and trace id.
  • Every outgoing reply has delivery proof or a visible failure.
  • Every short answer like "yes" is tied to a specific pending action in that specific person's thread.
  • No global mixed context. No borrowing context between teachers, parents, groups, or projects.
  • If Dorit cannot safely resolve meaning, it asks a clarification instead of guessing.

Sender: profile_noam

- Noam's private Telegram thread
- Noam's open pending actions

- The attached lead/invoice/calendar object

If exactly one pending action expects yes/no:
  execute that specific transition
Else:
  ask a clarification

6. Typed command proposal example

Natural language is allowed at the edge. After that, Dorit forces structure.

Input message:
"I am free now. Give me the new lead from Spain."

LLM may propose:
{
  "command_type": "accept_lead",
  "actor_sub": "profile_noam",
  "availability_update": "available_now",
  "lead_filter": { "country": "Spain", "status": "new" },
  "confidence": 0.94
}

Deterministic code must verify:
- actor_sub matches authenticated sender
- Noam has permission
- the Spanish lead exists and is uniquely resolved
- the lead is still available
- Noam is eligible
- no duplicate message/action exists
- action is allowed in current workflow state

7. Visual operations console: must exist from v0

This is not a later dashboard. The visual console is part of Dorit's safety system.

Goal: Zuriel can see everything live: who logged in, what request arrived, what context was injected, what command was proposed, what code validated, what workflow ran, what external API was called, what reply was sent, and what failed.

Events visible on the screen

  • Teacher logged into phone.zurot.org.
  • Parent sent WhatsApp question about invoice/payment.
  • Telegram teacher replied "yes" to lead offer.
  • Dorit linked external channel identity to ZurOt profile.
  • LLM proposed a typed command.
  • Schema passed or failed.
  • Permission check passed or failed.
  • Approval requested from admin.
  • PayPal invoice inspected, modified, or blocked.
  • Phone call started, connected, ended, and outcome recorded.
  • Facebook/Meta lead imported or assigned.
  • Calendar event proposed or created.

PayPal safety rules

  • Read-only invoice/payment answers can be allowed for verified users and matching household records.
  • Invoice modifications require admin approval unless explicitly scoped otherwise.
  • Recurring PayPal series are not safely editable through API; Dorit may report/find them and route admin to UI.
  • Every PayPal action requires idempotency key and visible proof.

11. First vertical slice

Build the smallest full system that proves the architecture.

  1. Telegram or web message arrives: "I am available".
  2. Sender is linked to ZurOt profile via OIDC linking.
  3. Dorit creates event and trace.
  4. LLM proposes update_availability.
  5. Schema validates.
  6. Permission validates: user can update own availability.
  7. State changes deterministically.
  8. Dorit replies with confirmation.
  9. Operations screen shows every stage live.

Second vertical slice

  1. Lead enters from Meta/Facebook or manual source.
  2. Dorit finds eligible available teacher.
  3. Dorit sends lead card to teacher.
  4. Teacher replies yes/no or clicks button.
  5. Pending action resolves exact meaning.
  6. Teacher opens Phone call link.
  7. Phone app emits call events.
  8. Dorit updates lead status and screen.

14. What not to do

  • Do not let the LLM directly execute PayPal, calendar, Facebook, or lead actions.
  • Do not use Telegram/WhatsApp identity alone for permissions.
  • Do not build one giant mixed chat memory.
  • Do not hide worker activity only in terminal logs.
  • Do not treat successful process/poller status as successful user outcome.
  • Do not allow retries to duplicate calls, lead assignments, or invoice changes.

15. Acceptance criteria for Dorit v0

  • A user can link Telegram or web chat to a ZurOt profile.
  • Dorit can receive a natural language availability message.
  • Dorit creates a typed command proposal.
  • Code validates, authorizes, and updates state deterministically.
  • A confirmation reply is sent and delivery/result is recorded.
  • The operations screen shows the full trace from inbound to final outcome.
  • A plain "yes" is resolved only through that user's pending action, or Dorit asks for clarification.
  • All events have trace IDs and idempotency keys.

18. Professional verdict

This can work. The vision is correct if Dorit is designed as an observable deterministic workflow system, not as a hidden free-agent chat bot.

The fastest safe path is a narrow prototype: ZurOt identity + Telegram/web input + availability workflow + visual trace screen. After that, add lead handoff and Phone calling. Then add PayPal/CRM read-only, and only later approval-gated financial actions.
